Byzantine Generals Problem Part 2

Posted on Wed 18 July 2018 in Mathematics, Programming, Distributed

Byzantine General's Problem Part 2¶

In this post, we're going to explore the one of the original papers by the authors of the Byzantine Generals Problem (Pease, Shostak, Lamport). The paper will be linked below. This is one of the seminal pieces that introduces the concept of fault tolerance in distributed systems, and sowed the seeds for practically Byzantine fault tolerant systems, one of which has become bitcoin. This part will speak specifically about the case of a single fault in a distributed network.

Reaching Agreement in the Presence of Faults- Introduction¶

The abstract/summary of this paper considers the problem associated with a set of isolated processors communicating with each other using only "two-party messages" (two party refers to the fact communications are between individual nodes e.g. P2P. there is no central node authority that send information to every other node simultaneously e.g. server-client). Each processor holds a value that has to be communicated to every other processor and then some consensus about the state of these values has to be reached for the entire distributed system to move forward. In the case that some of these processors are faulty, it is assumed that they can lie/miscommunicate the value being communicated to all the other processors, and this paper attempts to set a limit for the number of processors that can be faulty in order for the overall network to still function. As previously mentioned in part 1, distributed fault tolerant systems need some sort of internal messaging system in order to reach consensus.

Some Ideas¶

Lamport et al start by giving an example of a distributed systme needing consensus:

It may be necessary, for example,for the processors of a redundant system to synchronize their internal docks periodically. Or they may have to settle upon a value of a time-varying input sensor that gives each of them a slightly different reading. In the absence of faults reaching a satisfactory mutual agreement is usually an easy matter. In most cases it suffices simply to exchange values (times, in the case of clock synchronization) and compute some kind of average.

Lamport et al make it clear that in the case where all processors/nodes are honest, consensus is not an issue. But "in the presence of faulty processors, however, simple exchanges cannot be relied upon; a bad processor might report one value to a given processor and another value to some other processors, causing each to calculate a different average value" (Lamport et al 1)

It is possible to devise voting schemes, with multiple rounds where processors exchange values, in which honest processors can reach consistency and maintain the network by either identifying which processors are faulty and removing them from the network, or finding a way to work with the faulty processors by allowing all the nonfaulty processors to reach an exact agreement about network state. However it is not always possible to design these sorts of voting schemes, even with the number of faulty processors is known to be less than the number of honest processors. But algorithms that can allow exact agreement between honest processors, with a certain hard limit on the number of faulty processors in the network. The role of this paper is to establish a proof that gives a limit on the number of faulty processors that can exist in a network while it remains fault tolerant. (Spoiler: its proof by induction :) )

The Notion of Interactive Consistency¶

Consider a set of n isolated processors, of which it is known that no more than m are faulty (m < n). We are, however, not aware of which of the n processors are actually faulty. Furthermore, we assume that the processors can only communicate through two party messages. In this communication protocol, the messages are of negligible delay, and are fail-safe. That is to say, we assume integrity of the communication channel Along with that, the sender of the message is always identifiable by the reciever. That is to say, that processors cannot anonymously send messages in this communication protocol. In this situation, say that each processor has a private value $V_p$ (that could be something like its clock value as previously quoted, or some reading on an internal sensor as Lamport et al give as an example).

The question then becomes whether or not we can devise a message exchange algorithm such that for some number of total processors $m$, n $\geq$ 0, that will allow each honest processor $p_n$ to compute a vector of values with an element for each of the n processors. This scenario must satisfy the two following conditions.

  • 1) Each honest processor must compute the exact same vector.
  • 2) the element of each processor $p_n$ must be the private value of that processor

Such an algorithm would achieve interactive consistency, as it allows the honest (nonfaulty) processors to come to a consistent view of the private values of each processor, including the faulty ones. This vector is called the interactive consistency (IC) vector.

The framing of this scenario is meant to exemplify a consensus of state. Once the IC vector has been calculated, any number of functions can be applied to it (based on the goals of the application). We can ensure consensus however, because by definition, each processor will have the same IC vector, which guarantees that the system has a consistent state.

Lamport et al will spend the rest of this paper proving that any algorithm that guarantees interactive consistency can only do so for n,m such that n $\geq$ 3m +1. This means that for a case of a single faulty processor, a distributed system needs a minimum of 4 total processors in order to achieve interactive consistency (and thereby fault tolerance).

As an aside, it can be assumed that interactive consistency can be assured for any arbitrary $n \geq m \geq 0$ if we change the communication protocol in a way that ensures that faulty processors can refuse to pass on information obtained from other processors but cannot report false information (i.e. are no longer capable of being dishonest). This practically can be done using authenticators, which I will discuss later on in a separate post. For right now we focus on the case where we have no control over what faulty processors do.

Single Fault Case¶

This is a procedure for a single fault case: $m=1$ and $n=4$. This messaging procedure consists of one round of a message exchange where the processors all exchange their private values $V_p$. That is to say, in the situation of 4 processors, processor 1 would exchange its $V_p$ with processor 2,3,4, then the same would occur for processor 2, and so on. This is followed by a second round of messages where the processors exchange all the values they hold. After this the IC vector is computed and shared among all processors.

In the first round of this exchange, if a nonfaulty processor $p$ fails to recieve a message from some other processor $q$, it just chooses a value at random and assigns to the value they should have recieved from $q$.

In the second round of messaging, if the processors cannot agree on a common value for a specific processor (which would be the case if all of them had picked random values), then they just choose a NIL value to fill it with. This allows them to create an interactive consistency vector that they can all agree on. E.g. if Processor 2 had sent them all different values, then during round 2 they would realize that they could not agree on a shared value for it, and would set its $V_p$ as NIL or something of the sort.

Why does this assure Interactive Consistency?¶

We will explain this case by case.

No faulty Processors¶

In this case it should be fairly obvious that at the end of round one, all the processors have reached exactly the same conclusions about each other's private values. In round two, they confirm this and as a result, we can be certain that the IC vector will be exactly the same for all of them, and system state will be held.

Message Exchange- No Faulty Processors¶

In this first round, each processor exchanges its own $V_p$ value with every other processor. In the single fault case, this means that each of the 4 processors has 3 values from the other processors and its own private value. For example, that means that the 4 values that will be held by processor 1 are: $V{p1}$, $V{p_2}^2$, $V{p3}^3$, $V{p_4}^4$. We are using superscript here to signify which processor the private value has been recieved from. In this case processor 1 has held its own private value, and has recieved the other 3 from each processor respectively. We will use superscript to delineate how the private value changes as the message rounds proceed. Below I will utilize arbitrary numbers for each private value as I believe it will make this process more clear.

Message Exchange- Single Fault¶

We now repeat the code again in order to show what happens when there is a single processor being faulty. In a case where the faulty processor does not send any value, this is very simple. The processors all record NIL in round one, and proceed to round two very similarly to the above case. They would all have $V_{p2}$ = NIL for their IC vector.

However, let us consider the more difficult case where this faulty processor is malicious, and is communicating different values to each processor than its own actual internal value. I have picked 12 and 14 as the arbitrary values that it sends out to each other processor. There are other variants of this sort of fault, but we will demonstrate that the end result will be similar.

This code (very crudely) performs a check to imitate a processor checking to see if its vector matches the other vectors. It sees that it doesn't. All of the other processors would do this, and they would all end up changing their private values for processor 2 to NIL (or some other predefined value) as they would all reach the conclusion that their private value for 2 is different from everyone else's independently (while everything else is the same). As a result, they would then end up with the same interactive consistency vector. This interactive consistency vector would include a NIL value for processor number 2, which is critical to maintaing fault tolerance, as all the other three processors could then "excommunicate" processor 2 from the system. The critical point here is that while the 3 nonfaulty processors are not sure of the value of $V_{p2}$, they are all sure that processor 2 is malfunctioning, simply because they can agree to set its value to NIL. It is only because they all agree that their valuation of processor 2 in the secound round of message exchange is "fishy" that they can set it to NIL. This is the reason that multiple faults with only 4 processors will not work. It is like a majority opinion consensus. The 3 nonfaulty processors agree that their vectors match everything except values for processor 2. This is what allows them to establish a consisent system state (us 3 are good ,and processor 2 is faulty, since us 3 are in consensus about everything)

Multiple Faults with 4 processors.¶

The system would not be able to agree on a consistent state. As you can see above, the only reason that this situation is interactively consistent is because each of the nonfaulty processors is able to agree about everything except one value, which then allows them to excommunicate it. In a situation where multiple processors out of 4 were failing, the processors would not be able to even agree on the NIL value. Below we will make this clear by making both processor 2 and processor 3 faulty and running the same message exchange protocol again.

For processor one to be certain that a processor is faulty, it has to have consensus with all the other nonfaulty processors (2 of them) about the system state. That is the only way for it to mark it as NIL, In this case the processor cannot establish a majority about any of the states. For processor 2 there are 4 different values, but this is the case for processor 3 also. The key thing to remember here is that if the nonfaulty processors together cannot establish which values to make NIL, then the system cannot be interactively consistent. Processor 1 cannot know just from the information it has that Processor 4 is nonfaulty, as the two of them together cannot establish that both processor 2 and 3 are faulty (unlike the case above where all 3 nonfaulty processors could identify the faulty one and agree to call it NIL). It would require an outside party (authenticator, some sort of master node/server that resets the system). Thus the best it can do is delink itself from the network, and the system fails.
